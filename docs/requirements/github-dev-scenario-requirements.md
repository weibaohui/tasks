# GitHub 开发协作心跳场景需求文档

## 背景

在基于 GitHub 的软件开发流程中，从需求录入到代码合并涉及多个重复性环节：Issue 分析、代码编写、PR 评审、问题修复、文档补充、测试补齐等。目前这些环节完全依赖人工检查和处理，容易遗漏且效率低下。

为了让项目能够"自运转"，需要引入一套预定义的**心跳场景**（Heartbeat Scenario）。场景是一组协同工作的心跳集合，按固定周期持续驱动各个开发环节。人类通过打标签（如 `lgtm`）参与关键决策节点，保留最终控制权。

## 目标

1. 引入**心跳场景**概念：将一组相关心跳封装为可复用的场景模板。
2. 提供**GitHub 开发协作场景**：覆盖从 Issue 发现到 PR 合并的完整工作流。
3. 支持在项目配置中**一键选择并应用场景**，自动为项目注册场景内的所有心跳。
4. 场景应用后可独立启停、修改单个心跳，不影响其他项目。

## 功能需求

### FR-1 场景管理
- 系统内置 `github_dev_workflow` 场景，不可删除。
- 场景包含名称、描述、以及一组心跳模板定义（名称、间隔、Prompt、Agent、需求类型）。
- 提供场景列表查询接口。

### FR-2 项目场景绑定
- 项目编辑界面支持选择当前项目要使用的**心跳场景**（单选，可为空）。
- 选择场景后，系统为项目创建该场景定义的所有心跳实例。
- 已绑定场景的项目再次切换场景时，先清空旧场景产生的未修改心跳，再创建新场景心跳（简化策略：仅自动清理系统生成的心跳）。
- 场景应用后生成的心跳与普通心跳完全等同，支持独立编辑、启停、删除。

### FR-3 GitHub 开发协作场景心跳集

场景包含以下 8 个心跳，共同构成完整的开发协作流水线：

| 序号 | 心跳编码 | 心跳名称 | 默认间隔 | 职责描述 |
|------|----------|----------|----------|----------|
| 1 | `github_issue_analyzer` | Issue 分析 | 30 分钟 | 扫描项目 open issues，结合代码库分析，将分析结论评论到 issue 下。 |
| 2 | `github_lgtm_coder` | LGTM 代码编写 | 30 分钟 | 扫描带 `lgtm` 标签的 issue，拉取代码、实现需求、推送分支、创建 PR 并关联 issue。 |
| 3 | `github_pr_requirement_review` | PR 需求评审 | 30 分钟 | 扫描 open PRs，确认是否已进行"需求评审通过"评论；若无，追溯关联 issue 并在 PR 下补充需求评审结论。 |
| 4 | `github_pr_code_review` | PR 代码质量评审 | 30 分钟 | 扫描 open PRs，进行代码质量和安全评审，提出修改建议。 |
| 5 | `github_pr_fixer` | PR 修改修复 | 30 分钟 | 扫描自己提出的有修改建议的 PR，按建议修复代码并更新 PR。 |
| 6 | `github_pr_merge_check` | PR 合并检查 | 30 分钟 | 扫描 open PRs，判断是否达到可合并状态；若可以，在 PR 下回复 `/lgtm`。 |
| 7 | `github_pr_doc_writer` | PR 文档补充 | 60 分钟 | 扫描已合并或即将合并的 PR，根据代码变更补充/更新项目文档。 |
| 8 | `github_pr_test_writer` | PR 测试补充 | 60 分钟 | 扫描 open PRs，根据本次修改补充测试用例。 |

### FR-4 心跳 Prompt 规范
- 每个心跳的 Prompt 必须明确告知 Agent：
  - 使用 `gh` CLI 操作 GitHub。
  - 项目仓库地址为 `${project.git_repo_url}`。
  - 执行任务后给出明确的行动结论（做了什么、结果如何）。
  - 如果没有符合条件的条目，直接报告"无待处理项"，不执行任何操作。

## 非功能需求

- 场景数据与心跳数据分离：场景是模板，心跳是实例。
- 数据库变更向后兼容：新增 `heartbeat_scenarios` 表，修改 `projects` 表增加 `heartbeat_scenario_code` 字段。
- 核心 domain/application 逻辑单元测试覆盖率不低于 80%。
- 后端编译通过，`go vet` 无警告。
