# GitHub 开发协作心跳场景稳定性分析与优化设计

## 一、现状分析

当前内置的 `github_dev_workflow` 场景包含 8 个心跳项，默认间隔如下：

| 序号 | 心跳项 | 当前间隔 | 类型 |
|------|--------|----------|------|
| 1 | Issue 分析 | 30 min | 扫描+分析 |
| 2 | LGTM 代码编写 | 30 min | 行动（写代码） |
| 3 | PR 需求评审 | 30 min | 扫描+评审 |
| 4 | PR 代码质量评审 | 30 min | 扫描+评审 |
| 5 | PR 修改修复 | 30 min | 行动（改代码） |
| 6 | PR 合并检查 | 30 min | 扫描+检查 |
| 7 | PR 文档补充 | 60 min | 行动（写代码） |
| 8 | PR 测试补充 | 60 min | 行动（写代码） |

### 1.1 存在的问题

#### 频率过高，成本失控
- 8 个心跳每天触发约 200-400 次（假设 24h 运行）。
- 每个心跳都会调用 `gh` CLI、拉取仓库、调用 LLM，产生显著的 GitHub API 消耗和 Token 费用。
- 对于无活跃事务的时段，这些心跳在做大量"空转"扫描。

#### 缺乏阶段隔离，可能互相干扰
- PR 代码质量评审（第 4 步）会评审**所有 open PRs**，不判断是否已经通过需求评审。
- PR 修改修复（第 5 步）可能在评审意见刚发布后就立即执行，没有给人类审阅者留出确认时间。
- 文档补充和测试补充是**高成本的代码写入操作**，每小时执行会导致反复 push。

#### 缺乏去重与冷却机制
- 现有 Prompt 没有明确要求"检查自己最近是否已经处理过"。
- 同一个 issue 可能在一天内被分析 48 次，同一个 PR 可能被反复评审。
- 这会导致 GitHub 评论区被 AI 评论刷屏，降低可维护性。

#### 行动类任务频率与人力节奏不匹配
- 代码编写、修复、文档/测试补充需要编译、运行测试、验证结果。
- 30-60 分钟一次意味着 AI 可能在人类还没看完上一次变更时，就推送了新的 commit。

## 二、稳定性控制策略

### 2.1 频率分层原则

根据任务性质划分为三层频率：

| 层级 | 任务类型 | 建议间隔 |  rationale |
|------|----------|----------|-----------|
| **高频扫描** | 状态检查、合并检查 | 2h | 需要较快感知状态变化，但无需实时 |
| **中频评审** | Issue 分析、PR 各类评审 | 3h | 给人类留出阅读和响应时间 |
| **低频行动** | 代码编写、修复、文档/测试补充 | 4-8h | 写入操作成本高，且需要与人工节奏对齐 |

### 2.2 阶段闸门（Stage Gating）

通过 Prompt 中的前置条件过滤，确保下游步骤只处理上游已完成的项：

```
Issue 分析 → LGTM 标签 → 代码编写
                ↓
            PR 创建 → 需求评审 → 代码质量评审 → 修改修复 → 合并检查
                ↓
            文档补充 / 测试补充（仅在代码稳定后）
```

具体闸门规则：
- **代码质量评审**：仅处理已标记"需求评审通过"的 PR。
- **修改修复**：仅处理存在未解决评审意见且**最近一次评审意见超过 2 小时**的 PR。
- **合并检查**：仅处理存在 `/lgtm` 或类似批准标记的 PR。
- **文档/测试补充**：仅处理最近 24 小时内有代码 push 的 PR，且避免与"修改修复"同时活跃。

### 2.3 冷却与去重机制

每个 Prompt 必须增加以下约束：
- **时间冷却**："检查该 issue/PR 最近 3 小时内是否已有你的评论，如有则跳过。"
- **数量上限**：扫描类每次最多处理 N 个（已存在），行动类每次**严格只处理 1 个**。
- **空转即退**：如果没有符合条件的条目，必须在第一步就返回，不执行 clone/分析/评论。

### 2.4 自我修复与收敛

系统应趋向稳定而非振荡：
- 评审意见发出后，**不立即修复**，留出冷却期。
- 修复完成后，**不立即再次评审**，让下一轮正常心跳去处理。
- 文档/测试补充在代码写入任务完成后的一段时间再进行，避免并发修改同一分支。

## 三、优化后的时间配置

| 序号 | 心跳项 | 优化后间隔 | 优化原因 |
|------|--------|-----------|----------|
| 1 | Issue 分析 | **180 min** | issue 不会频繁新增，3h 足够及时 |
| 2 | LGTM 代码编写 | **120 min** | 需要完整编码时间，且依赖人类打标签 |
| 3 | PR 需求评审 | **120 min** | 给 PR 创建者留出完善时间 |
| 4 | PR 代码质量评审 | **180 min** | 应在需求评审之后，降低频率避免重复评审 |
| 5 | PR 修改修复 | **240 min** | 4h 冷却，给人类确认和讨论留出时间 |
| 6 | PR 合并检查 | **120 min** | 轻量检查，保持适中频率 |
| 7 | PR 文档补充 | **480 min** | 每天约 3 次，写入类任务不宜过频 |
| 8 | PR 测试补充 | **480 min** | 每天约 3 次，与文档补充错开执行 |

**整体效果**：每日总触发次数从约 300 次降至约 60 次，降低 **80%** 的空转和 API 消耗。

## 四、Prompt 通用稳定性约束（应加入所有心跳项）

1. **仓库定位**：使用 `gh` CLI，通过 `git remote get-url origin` 或项目配置获取 owner/repo。
2. **时间冷却**：处理任何 issue/PR 前，先查看该 issue/PR 的最近评论，若你在 **3 小时内**已评论过，跳过该条目。
3. **批次限制**：扫描类最多处理 2-3 个，行动类**严格只处理 1 个**。
4. **状态前置**：
   - 代码编写仅处理带 `lgtm` 标签的 open issue。
   - 代码质量评审仅处理已存在"需求评审通过"评论的 PR。
   - 修改修复仅处理存在未解决评审意见的 PR。
   - 合并检查仅处理已存在 `/lgtm` 且 CI 通过的 PR。
5. **无任务即退**：若列表为空或全部因冷却/状态不满足而被跳过，立即返回"当前无待处理项"。
6. **不源代码破坏性操作**：不执行 `gh pr merge`，不删除分支，不关闭 issue（除非通过 PR 描述自动关联）。

## 五、实施建议

### 5.1 短期（本次优化）
- 调整 8 个心跳项的 `IntervalMinutes` 为优化值。
- 在 Prompt 中增加"3 小时冷却检查"、"每次最多 1 个"、"状态前置"等约束。

### 5.2 中期（后续迭代）
- 在 `heartbeats` 表中增加 `last_processed_at` 和 `processed_items` 字段，由系统层面实现去重，而非仅靠 Prompt。
- 为 PR 和 issue 引入本地状态缓存（如 `heartbeat_logs` 表），记录 AI 已处理过的对象及时间戳。

### 5.3 长期
- 将 8 个心跳合并为 1 个"工作流编排器"心跳，统一调度 8 个子任务，避免各自扫描和时序冲突。
- 引入状态机驱动：每个 issue/PR 作为一个需求进入状态机，心跳只负责状态推进而非重复扫描。
